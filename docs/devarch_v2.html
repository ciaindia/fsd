GitHub Pull Request Workflow with Jenkins CI/CD

üìã Prerequisites

Before starting with this workflow, ensure all team members have the following set up properly.
  • Git Installation: Git must be installed on all developer machines
    # Check if Git is installed
    git --version

    # Download from: https://git-scm.com/downloads
  • GitHub Account: Each developer needs a GitHub account (https://github.com/signup)
  • Repository Access: All developers must have write access permissions to the repository
  • Git Configuration: Configure user name and email
    git config --global user.name "Your Name"
    git config --global user.email "your_email@example.com"
  • GitHub Personal Access Token: Create a token for HTTPS authentication
    # Generate token from:
    # GitHub ‚Üí Settings ‚Üí Developer settings ‚Üí Personal access tokens
    # Select scopes: repo (full control of private repositories)

    # Store credentials to avoid repeated login
    git config --global credential.helper store

üìñ Overview

This architecture demonstrates a GitHub Pull Request-based workflow where developers work on their own branches, push changes using HTTPS, and create pull requests for code review. The maintainer reviews and merges changes through the GitHub portal. The system uses the dev branch for integration, main/master branch for releases, and Jenkins for automated deployment.
GitHub Repository Pull Requests Jenkins CI/CD UAT Environment Production Environment

üîÄ Complete Workflow Process

1. Clone Branch

  • Each developer clones their specific branch
  • Use HTTPS URL for cloning
  • Work on backend and frontend code

2. Make Changes & Push

  • Make code changes in local branch
  • Commit with meaningful messages
  • Push changes to GitHub

3. Create Pull Request

  • Go to GitHub repository
  • Create PR from your branch to dev
  • Add description of changes
  • Submit for review

4. Code Review

  • Maintainer reviews PR on GitHub
  • Reviews code changes and comments
  • Developer addresses feedback
  • Maintainer approves when satisfied

5. Merge to Dev

  • Maintainer merges PR
  • Changes merged to dev branch
  • Optional: Delete feature branch

6. Release Process

  • Create PR: dev to main
  • Final review of all changes
  • Merge to main/master
  • Jenkins auto-deploys

üìö Developer Guide - Step by Step

Step 1: Clone Your Branch

# Clone your specific branch using HTTPS
git clone -b dev1 https://github.com/organization/anyproject

# Navigate to directory
cd anyproject

# Verify branch
git branch

Step 2: Make Changes and Commit

# Check status
git status

# Add files
git add .

# Commit
git commit -m "Added new feature"

Step 3: Push to GitHub

# Push changes
git push origin dev1

# First time: enter username and Personal Access Token

Step 4: Create Pull Request

On GitHub web interface:

  1. Go to repository on GitHub
  2. Click "Pull requests" tab
  3. Click "New pull request"
  4. Set base: dev, compare: dev1
  5. Add title and description
  6. Click "Create pull request"

üë®‚Äçüíº Maintainer Guide

Review and Merge Pull Requests

  1. Go to repository Pull requests tab
  2. Click on PR to review
  3. Review "Files changed" tab
  4. Add comments if needed
  5. Click "Review changes"
  6. Select "Approve" or "Request changes"
  7. Click "Merge pull request" when ready
  8. Confirm merge

Release to Production

  1. Create new PR: dev ‚Üí main
  2. Review all accumulated changes
  3. Add release notes
  4. Merge to main branch
  5. Jenkins auto-deploys to UAT then Production

üîß Troubleshooting

‚ùå Authentication Failed

Problem: Cannot push due to authentication error

Solution:

Use Personal Access Token instead of password

# Create token: GitHub ‚Üí Settings ‚Üí Developer settings
# ‚Üí Personal access tokens ‚Üí Generate new token
# Use token as password when pushing

‚ùå Failed to Push - Remote has Changes

Problem: Remote branch has updates you don't have

Solution:
git pull origin dev1
# Resolve conflicts if any
git push origin dev1

‚ùå Merge Conflicts

Problem: Conflicts when pulling or merging

Solution:
  1. Open conflicted files
  2. Look for conflict markers (<<< === >>>)
  3. Choose which code to keep
  4. Remove conflict markers
  5. git add . && git commit
  6. git push origin dev1

‚úÖ Best Practices

For Developers

  • Write clear commit messages
  • Commit frequently
  • Test before pushing
  • Keep PRs focused and small
  • Sync with dev branch daily
  • Never commit sensitive data

For Maintainers

  • Review code thoroughly
  • Provide constructive feedback
  • Ensure tests pass
  • Check for code standards
  • Verify documentation
  • Test before releasing
